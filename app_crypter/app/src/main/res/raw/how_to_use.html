How to use ?

Encrypt

  • select one file or write some text
  • select a key alias or enter a password
  • click Encrypt button
  • when the encryption is done, you can share the encrypted file or the encrypted text

Decrypt

  • select one encrypted file or paste encrypted text
  • select a key alias or enter a password used for the encryption
  • click Decrypt button
  • when the decryption is done, you can open the decrypted file or read the decrypted text

Keystore

Instead using a disposable password each time, the user can use the Keystore in order to generate keys linked to an alias. Then to encrypt or to decrypt, the user just need to select an alias.

The Keystore is protected by an user password. All keys generated are retained encrypted in the device.

By default, the password used to generate one key is not retained.

If the user prefer to share its passwords instead of the keys with another device, then the password can be retained encrypted with a non reversible encryption method. This allow one device which do not own the key used to encrypt to be able to decrypt only with the password used to encrypt it.

In order to retain the password (non reversible encrypted), the user must uncheck himself the checkbox Nullify Password when creating a new key.


Share a key

The sharing is done with a 2 steps protocol started by the receiver. The receiver need to send one public key to the sender. Then the sender encrypt his key and send it to the receiver.

  • 1- Receiver open the Keystore (key button)
  • 2- Receiver click on the action(+) button (top right)
  • 3- Receiver click Import one shared key button
  • 4- Receiver send the public key to the Sender (with one messaging application or just show the qr code to be scanned by the sender)

  • 5- Sender open the Keystore (key button)
  • 6- Sender slide right and click the share button of the key that you want to share
  • 7- Sender retrieve the public key from the Receiver (from one messaging application or just scan the qr code presented by the receiver)
  • 8- Sender send the encrypted key to the Receiver (with one messaging application or just show the qr code to be scanned by the receiver)

  • 9- Receiver receive the encrypted shared key from the Sender (from one messaging application or just scan the qr code presented by the sender)
  • 10- Receiver click on the Confirm button to confirm the import of the key

Strict mode

By default, for each encryption the salt, the iteration number, the key length and the encryption algorithm are appended to the encrypted data in order to rebuilt the key from the password. Therefore you can decrypt the data with the password or the key

The user can also activate the option Strict mode. With this mode, no information to rebuild the key from the password are appended to the encrypted data. Only the key can decrypt the data (In order to add another protection level against any dictionaries attacks). With this mode, it is very important that the user do a backup of his keys with Export all keys features and keep them in safe place.

Lose these keys and it will be impossible to decrypt your data


Notes

-Encrypted files/texts will have ".cfx" or ".ctx" extension and will be in android media folder Documents or inside the user defined Directory

-Each encryption is done with an AES key randomly generated from the password or the key bound to an alias

-Decrypted files will be in your android media folder (Pictures, Music, Movies or Documents) or inside the user defined directory

-Decrypted files will have the original file name maybe with a copy number if one file with this name exist already


How Encryption is done?

-Encryption is done with the method chosen by user (AES_CTR, AES_CBC, AES_GCM). Recommended method is AES_GCM. Other method (AES_CTR, AES_CBC) are also secure enough and little bit faster

-Each Secret key is derived from password with PBKDF2WithHmacSHA1 algorithm

-Each Secret key length is chosen by user (128, 192, 256). Recommended length are 256 even if (128, 192) are as much secure

-Each iteration count and salt are randomly generated for each new Secret key with the algorithm SHA1PRNG

-Encrypted data are signed with HmacSHA256 algorithm

-For each encryption a new randomly IV generated with the algorithm SHA1PRNG is used

-For each Signature a new randomly IV generated with the algorithm SHA1PRNG is used along the tag

-Only for the encryption of text, the user can chose an encryption without signing and authentication in order to reduce the length of the message. It is also secure enough, but we recommend to let the signing and authentication. If the message length is a problem for you, just share your encrypted text as file.

-Sharing a key with is done with a pair of asymmetric keys generated on both device then send with ECDH protocol
